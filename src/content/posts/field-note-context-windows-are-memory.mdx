There was a period — roughly 2023 into 2024 — when prompt engineering was the primary skill people associated with effective AI use. Craft the right instruction, elicit the right behavior. That framing made sense when most AI interaction was single-shot: one prompt, one response.

Agentic workflows break that model immediately.

An agent working on a non-trivial task operates across many steps. At each step, its context window is its working memory — everything it can reason about right now. What goes into that window is not just the current prompt. It is the accumulated history of the session, the loaded files, the tool outputs, the injected documentation. The agent cannot reason about what is not in the window. It cannot remember what fell out of it.

**This means the quality of agent output is not primarily a function of how well you prompt — it is a function of what you have loaded into the working memory at the moment the agent acts.**

This is a practical constraint with practical consequences.

When an agent starts implementing a feature and begins generating plausible but subtly wrong code, the failure mode is almost never a bad instruction. It is almost always a context gap — the agent lacks information it needs to reason correctly. The relevant architecture decision was not loaded. The convention was not documented. The interface contract was not in scope.

The response to that failure is not to rewrite the prompt. It is to fix the context.

Context engineering — the discipline of deciding what information gets loaded, in what form, and at what point in the agent's workflow — is the emerging practice that determines whether agentic workflows produce acceptable output at scale. It is less glamorous than prompt engineering. It is more consequential.

A few things that follow from this:

**Documentation is now runtime infrastructure.** Your CLAUDE.md, your ADRs, your convention files — these are not static reference documents. They are loaded into the context window and directly shape what the agent can reason about during execution. Low-quality documentation produces low-quality agent output. The causal chain is tight and fast.

**Context budget is a design constraint.** Every token in the context window is a token that could have contained something else. Verbose onboarding docs, long historical preambles, unfocused system prompts — these consume budget that could hold the specific knowledge the agent needs for the current task. Context engineering requires the same discipline as good technical writing: ruthless prioritization of signal over volume.

**Task scope is a context architecture decision.** When you decompose work into agent tasks, you are implicitly deciding what context will fit in the window together. A task that requires the agent to hold the entire system in mind simultaneously is a task that will fail — not because the agent is incapable, but because the working memory cannot hold the relevant state.

The teams producing consistently good agentic output are not necessarily using better models or better prompts. They are investing in context infrastructure — the structured, precise, well-maintained knowledge base that fills the working memory with the right information when it is needed.

Prompt engineering is necessary. Context engineering is where the leverage is.
